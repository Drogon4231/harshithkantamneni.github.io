Twelve cycles in a row, my lab's cross-platform build hashed byte-identically to the cycle before it. Five surfaces (Android arm64, armeabi-v7a, x86, x86_64, plus iOS Mach-O), all matching.

Sounds like nothing happened. It actually means something specific: across roughly two weeks of cycles, no specialist agent introduced an undeclared change to the build output. Every change that should have been visible would have shown up as a hash mismatch. None did.

Three things made the streak legible:

1. An R11 invariant that names exactly which surfaces must hash-match the prior cycle: per-architecture .so files, the iOS Mach-O, and the symbol-name set inside them.

2. A build-history TSV row appended at every cycle close, so the chain is grep-recoverable across hundreds of cycles.

3. A golden symbol parity check requiring hive_core_zeroize and hive_core_hkdf_sha256 to appear at the same address as the prior binary. Catches the case where the symbol set is intact but its layout shifted underneath.

The point isn't the bytes. It's that drift in the build is now legible to an outside observer. If public CI compiles the same source and produces a different hash, something in the build environment changed. If it produces an identical binary, the lab's claim that nothing meaningful changed this cycle is verifiable from outside the trust boundary.

Without that, a build-pass streak is just another self-reported number.
